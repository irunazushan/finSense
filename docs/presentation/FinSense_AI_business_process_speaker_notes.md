# FinSense: заметки к выступлению на 10 минут

## Общая логика доклада

Главный тезис: FinSense показывает не “чат с ИИ”, а внедрение LLM в конкретный backend-бизнес-процесс. LLM используется точечно: там, где классические правила дают низкую уверенность или где нужно сформировать понятную пользователю рекомендацию на основе фактов.

## Слайд 1. Название и тезис

Вступление: проект FinSense демонстрирует гибридную AI-систему для классификации банковских транзакций и генерации персональных финансовых рекомендаций.

Важно подчеркнуть, что LLM здесь не заменяет всю систему. Она встроена как отдельный интеллектуальный слой, который подключается только к сложным участкам процесса.

Ориентир по времени: 40 секунд.

## Слайд 2. Проблема бизнеса

Проблема состоит в том, что классическая категоризация по MCC-кодам и ключевым словам хорошо работает на типовых операциях, но ошибается на неоднозначных описаниях, новых мерчантах и противоречивых данных.

Вторая проблема — рекомендации. Без анализа поведения клиента они превращаются в шаблонные советы, которые пользователь быстро перестаёт воспринимать как полезные.

Третья проблема — стоимость. Можно отправлять каждую транзакцию в LLM, но это дорого, медленно и архитектурно неэффективно.

Ориентир по времени: 1 минута.

## Слайд 3. Идея решения

Решение — гибридный AI-пайплайн. Массовые типовые операции обрабатывает быстрый rule/ML-first слой. Если confidence выше порога, транзакция классифицируется сразу.

Если confidence ниже порога 0.9, Core Service публикует событие в Kafka, а LLM-агент асинхронно уточняет категорию. Это позволяет повысить качество на сложных случаях без роста стоимости на всём потоке.

Отдельно Financial Coach Agent использует LLM для формулирования рекомендаций.

Ориентир по времени: 1 минута.

## Слайд 4. Архитектура FinSense

В центре системы находится Core Service. Он принимает события транзакций, хранит доменную модель и оркестрирует процесс.

Classifier Service — быстрый stateless-сервис для первичной классификации.

Transaction Classifier Agent — LLM fallback для неуверенных транзакций.

Financial Coach Agent — аналитический LLM-агент для рекомендаций.

Kafka используется как шина событий, PostgreSQL — как хранилище транзакций, статусов, рекомендаций и AI-метаданных.

Ориентир по времени: 1 минута 15 секунд.

## Слайд 5. Поток классификации транзакции

Сырая транзакция попадает в topic `raw-transactions`. Core Service сохраняет её и вызывает Classifier Service.

Если confidence больше или равен 0.9, транзакция получает статус `CLASSIFIED`, а источник классификации сохраняется как `ML` или rule-based слой MVP.

Если confidence ниже 0.9, транзакция переходит в `LLM_CLASSIFYING`, а Core публикует запрос в `llm-classifier-requests`. После ответа агента Core обновляет категорию, confidence и статус.

Ориентир по времени: 1 минута 10 секунд.

## Слайд 6. Как LLM улучшает процесс

LLM полезна именно в “серой зоне”: когда описание неоднозначно, MCC-код конфликтует с текстом или merchant не попал в правила.

Агент получает текущую транзакцию, предсказанную категорию, confidence и историю операций пользователя. В ответ он возвращает только одну из разрешённых категорий, confidence и короткое reasoning.

Такой подход снижает долю `UNDEFINED` и потенциальную ручную проверку, но не блокирует быстрый путь для типовых операций.

Ориентир по времени: 1 минута 10 секунд.

## Слайд 7. Financial Coach Agent

Для рекомендаций используется ReAct-подход: LLM не выдумывает факты, а сначала получает данные через tools.

В проекте есть четыре инструмента: траты по категориям, месячная дельта, топ-мерчанты и детекция всплесков расходов.

LLM получает фактическую аналитику и формирует краткий совет на русском языке. Поэтому рекомендация становится персональной, но остаётся основанной на проверяемых данных.

Ориентир по времени: 1 минута 15 секунд.

## Слайд 8. Метрики эффекта

Метрики в презентации нужно трактовать как пилотную оценку MVP на синтетических данных и как методику измерения, а не как production-результат.

Ключевые метрики: точность классификации, доля LLM fallback, снижение доли ручной проверки или `UNDEFINED`, p95 latency, стоимость на 1 млн транзакций, полезность рекомендаций и наблюдаемость AI-решений.

Основной экономический смысл: LLM не применяется к 100% транзакций. Она вызывается только для 5-10% сложных случаев, поэтому качество растёт без пропорционального роста стоимости.

Ориентир по времени: 1 минута 20 секунд.

## Слайд 9. Итог и переносимость

Завершение: этот бизнес-кейс синтетический. Он создан не для запуска реального банковского продукта, а для демонстрации того, как LLM может быть встроена в бизнес-процесс.

Главная ценность проекта — архитектурный паттерн: быстрый детерминированный слой, confidence gate, LLM для сложных случаев, асинхронность, аудит и сохранение результата.

Этот подход переносим на другие процессы. Примеры: страхование для первичной обработки заявлений, кредитование для предварительного анализа заявок, support для маршрутизации обращений, документооборот для анализа договоров и счетов.

Финальная мысль: LLM даёт максимальный эффект там, где есть неоднозначность, контекст и высокая стоимость ручного решения.

Ориентир по времени: 1 минута 10 секунд.
