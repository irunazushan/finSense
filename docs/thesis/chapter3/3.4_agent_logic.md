# 3.4. Реализация агентной логики и интеграция генеративных компонентов

## Назначение подраздела

Подраздел описывает LLM-компоненты прототипа: Transaction Classifier Agent и Financial Coach Agent, их роли, входные данные, промпты, обработку ответов и взаимодействие с остальной системой.

## Черновая структура

### 3.4.1. Общая роль LLM-компонентов

- Обработка случаев, где требуется контекстная интерпретация.
- Асинхронное выполнение тяжёлых операций.
- Разделение ответственности между двумя агентами.

### 3.4.2. Transaction Classifier Agent

- Получает события `llm-classifier-requests`.
- Использует данные текущей транзакции и историю пользователя.
- Формирует prompt для LLM.
- Возвращает категорию, confidence и reasoning.
- Публикует результат в `llm-classifier-responses`.

### 3.4.3. Financial Coach Agent

- Получает события `coach-requests`.
- Читает агрегированные данные по транзакциям пользователя.
- Использует deterministic tools для подготовки фактической аналитики.
- Формирует prompt на основе результатов tools.
- Сохраняет рекомендацию в PostgreSQL и публикует `coach-responses`.

### 3.4.4. Инструменты анализа расходов

- `getSpendingByCategory`.
- `getMonthlyDelta`.
- `getTopMerchants`.
- `detectSpikes`.
- Объяснить, почему LLM получает агрегированные данные, а не полный набор сырых транзакций.

### 3.4.5. Интеграция с LLM API

- Используемый провайдер: DeepSeek-compatible API.
- Параметры: model, temperature, max tokens, timeout.
- Обработка JSON-ответов.
- Повторные попытки при ошибках.

### 3.4.6. Логирование и аудит AI-решений

- Сохранение prompt/response.
- Метаданные: model, tokens, latency, success.
- Correlation identifiers: `transactionId`, `requestId`.
- Использование логов для отладки и будущей экспериментальной оценки.
