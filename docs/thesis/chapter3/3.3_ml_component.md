# 3.3. Подготовка данных и разработка ML-компонента для классификации транзакций

## Назначение подраздела

Подраздел описывает подготовку данных, обучение модели классификации транзакций, экспорт в ONNX и интеграцию ML-компонента в прототип.

## Черновая структура

### 3.3.1. Назначение ML-компонента

- Быстрая первичная классификация транзакций.
- Возврат категории и confidence.
- Снижение количества обращений к LLM.

### 3.3.2. Категории классификации

- `FOOD_AND_DRINKS`.
- `TRANSPORT`.
- `GROCERIES`.
- `RETAIL_SHOPPING`.
- `ENTERTAINMENT`.
- `HEALTH`.
- `BANKING_AND_FEES`.
- `BILLS_AND_GOVERNMENT`.
- `UNDEFINED`.

### 3.3.3. Подготовка данных

- Использование `tools/ml-training`.
- Генерация датасетов из `transaction_catalog.yaml`.
- Профили данных: balanced и realistic.
- Разделение на train/validation/test.
- Учет неоднозначных и низкосигнальных транзакций.

### 3.3.4. Обучение модели

- Описание Python training pipeline.
- Используемые признаки: сумма, описание, merchant name, MCC и производные признаки.
- Метрики качества на этапе обучения: accuracy, macro F1, weighted F1, confusion matrix.

### 3.3.5. Экспорт и runtime-интеграция

- Экспорт модели в `transaction-classifier.onnx`.
- Сохранение `labels.json` и `metadata.json`.
- Подключение артефактов в `classifier-service`.
- Выполнение inference через ONNX Runtime.

### 3.3.6. Роль confidence в гибридном пайплайне

- Confidence используется Core Service для выбора дальнейшего маршрута.
- Высокая уверенность: результат сохраняется как итоговая классификация.
- Низкая уверенность: транзакция отправляется в Transaction Classifier Agent.

## Открытые пункты

- После экспериментов добавить фактические значения метрик ML-модели.
- Уточнить финальный порог confidence, если он будет изменён перед экспериментом.
