# 3.5. Методика экспериментальной оценки прототипа

## Назначение подраздела

Подраздел будет описывать, как оценивается эффективность разработанного прототипа. Фактические значения метрик будут добавлены после проведения экспериментов.

## Черновая структура

### 3.5.1. Цель экспериментальной оценки

- Проверить качество классификации транзакций.
- Оценить вклад LLM fallback в гибридном пайплайне.
- Измерить задержку обработки.
- Оценить воспроизводимость и трассируемость процесса.

### 3.5.2. Сравниваемые режимы

- Базовый режим: только ML/ONNX-классификатор.
- Гибридный режим: ML/ONNX + Transaction Classifier Agent с retrieval-обогащением сложных случаев.
- Режим генерации рекомендаций: Financial Coach Agent на основе агрегированных данных.

### 3.5.3. Тестовые данные

- Balanced dataset.
- Realistic dataset.
- Низкосигнальные и неоднозначные транзакции.
- Holdout merchants для проверки обобщающей способности модели.

### 3.5.4. Метрики классификации

- Accuracy.
- Macro F1.
- Weighted F1.
- Precision/recall/F1 по категориям.
- Confusion matrix.
- Доля fallback-запросов к LLM.

### 3.5.5. Метрики производительности

- Latency ML-инференса.
- End-to-end latency классификации.
- p95 времени обработки.
- Количество обработанных сообщений.

### 3.5.6. Метрики агентного контура

- Доля успешных LLM-ответов.
- Количество повторных попыток.
- Latency LLM-вызовов.
- Корректность структуры ответа.
- Качество рекомендаций по выбранной экспертной шкале.

### 3.5.7. Инструменты проведения эксперимента

- `tools/ml-training`.
- `tools/transaction-tester`.
- `tools/prompt-lab`.
- Логи LLM-компонентов.
- Kafka-события и данные PostgreSQL.
