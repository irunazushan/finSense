# 3.1. Постановка задачи, описание частично формализуемого бизнес-процесса

В рамках данной работы рассматривается автоматизация обработки финансовых транзакций и формирования персонализированных рекомендаций для клиента банка. Сценарий выбран как пример бизнес-процесса, где часть операций хорошо описывается формальными правилами и структурами данных, а часть требует контекстной интерпретации. На этом разделении строится гибридный подход прототипа FinSense: быстрый ML-компонент выполняет первичную классификацию, а AI-агент на базе LLM подключается к случаям, где признаков транзакции недостаточно для уверенного решения.

Прототип ориентирован на два связанных процесса. Первый связан с классификацией банковских транзакций по категориям расходов. Второй использует уже классифицированную историю операций для подготовки персональных финансовых рекомендаций. В банковском или финтех-приложении ошибка на первом этапе быстро переходит во второй: неверно определённая категория ухудшает отчёты по расходам и снижает практическую ценность рекомендации.

## Общая постановка задачи

Основная задача прототипа состоит в построении системы, которая принимает поток транзакций, определяет категорию каждой операции и формирует рекомендации на основе накопленной истории расходов. При проектировании учитывалось, что транзакции различаются по сложности автоматической обработки. У части операций есть достаточно понятные признаки: сумма, MCC, merchant name, описание платежа. Для таких случаев используется ML-классификатор, который выполняет быстрый inference через ONNX Runtime и возвращает категорию вместе с оценкой уверенности.

Другая часть транзакций содержит неоднозначные или слабо информативные описания. Например, merchant name может быть сокращён, транслитерирован, содержать технический идентификатор терминала или название платёжного посредника вместо фактического продавца. В таких случаях одной модели классификации может оказаться недостаточно: требуется учитывать историю пользователя, соседние операции и смысловое содержание описания. Для этого в проекте используется Transaction Classifier Agent, который получает низкоуверенные случаи, а при недостаточности локального контекста дополнительно извлекает похожие сложные кейсы из `pgvector` и обрабатывает запрос с помощью LLM.

Генерация рекомендаций устроена похожим образом, но с другим акцентом. Агрегированные расходы по категориям вычисляются детерминированно, тогда как итоговый совет должен учитывать поведение пользователя и не сводиться к пересказу таблицы расходов. Financial Coach Agent использует инструменты анализа, получает фактические агрегаты из базы данных и передаёт их в LLM для формирования рекомендации. Качество такого ответа зависит от полноты исходных данных и корректности предварительных расчётов, поэтому LLM в данном контуре не рассматривается как самостоятельный источник финансовой аналитики.

## Описание бизнес-процесса

Обрабатываемый бизнес-процесс начинается с появления финансовой операции пользователя. В прототипе такие операции поступают в систему как события raw-transactions, что имитирует поток транзакций от внешней банковской системы или тестового генератора. Каждая транзакция содержит идентификатор, пользователя, счёт, сумму, описание, merchant name, MCC и дату операции.

После поступления события Core Service сохраняет транзакцию и запускает первичную классификацию. Для этого он синхронно обращается к Classifier Service, где выполняется ML-инференс через ONNX Runtime. Если возвращённое значение confidence достаточно высокое, результат классификации принимается как итоговый: транзакция получает категорию, источник классификации и статус завершённой обработки.

Если уверенность классификатора ниже заданного порога, Core Service не принимает результат как окончательный. Он публикует событие в llm-classifier-requests. Transaction Classifier Agent получает запрос, формирует prompt на основе данных текущей транзакции и контекста пользовательской истории. Для неоднозначных случаев агент выполняет retrieval похожих кейсов из `pgvector`, обогащает prompt и затем вызывает LLM, после чего публикует результат в llm-classifier-responses. После этого Core Service обновляет сохранённую транзакцию.

Отдельный поток связан с рекомендациями. Пользователь или планировщик инициирует запрос на получение финансового совета. Core Service создаёт запись рекомендации со статусом PENDING и публикует событие coach-requests. Financial Coach Agent читает транзакции пользователя, выполняет агрегирующие инструменты анализа и формирует рекомендацию с помощью LLM. Результат сохраняется в PostgreSQL, а событие coach-responses может использоваться Notify Service для отправки уведомления.

Роли компонентов в этом процессе различаются. Core Service выполняет оркестрацию и хранит состояние обработки. Classifier Service отвечает за быстрый ML-контур, Transaction Classifier Agent — за контекстную дообработку сложных транзакций с retrieval-обогащением через `pgvector`, Financial Coach Agent — за генерацию рекомендаций, а Kafka обеспечивает асинхронный обмен событиями между компонентами. Такое распределение удобно для прототипа, но оно также увеличивает требования к согласованности контрактов и трассировке сообщений между сервисами.

## Частично формализуемый характер процесса

Рассматриваемый процесс нельзя отнести ни к полностью формализованным, ни к полностью неформализованным. Его основа хорошо структурирована: у транзакции есть фиксированный набор полей, категории расходов задаются заранее, жизненный цикл обработки описывается конечным набором статусов, а взаимодействие сервисов строится на REST- и Kafka-контрактах. Эти элементы можно представить через схемы, DTO, таблицы базы данных и состояния.

Смысловая интерпретация финансовой операции не всегда сводится к значениям отдельных полей. Одна и та же строка описания может означать разные типы расходов в зависимости от merchant name, суммы, пользовательской истории и контекста. Новые продавцы, нестандартные названия, платёжные агрегаторы и неполные описания снижают качество прямой классификации. Здесь появляется слабо формализуемая часть процесса: системе нужно сопоставлять признаки и интерпретировать их с учётом окружения операции.

Генерация рекомендаций также сочетает формальные и слабо формализуемые элементы. Расходы по категориям, динамику за период и крупных мерчантов можно получить детерминированными запросами к данным. Итоговый совет не должен быть простой строкой из шаблона. Он должен учитывать, где пользователь действительно тратит больше, какие категории изменились за период и какой вывод можно сделать на основе подготовленных агрегатов. Поэтому в проекте LLM не заменяет аналитический расчёт, а используется после него, когда фактические данные уже получены инструментами.

Такое разделение отражает инженерную логику прототипа. Формализуемые части процесса реализованы через сервисные контракты, базу данных, ML-инференс и обработку событий. Слабо формализуемые фрагменты вынесены в AI-агенты, которые работают асинхронно и не блокируют основной поток обработки. При этом результат работы агентов требует контроля: возможны ошибки интерпретации, зависимость от качества prompt и нестабильность ответа при изменении входных данных.

## Границы автоматизации в прототипе

В прототипе автоматизируется полный путь от поступления транзакции до получения классифицированной операции и рекомендации. Система не ограничивается отдельным ML-классификатором: в неё включены fallback-классификация и агентный контур финансового советника.

Классификация транзакций выполняется в два этапа. На первом этапе применяется ML/ONNX-компонент, рассчитанный на быстрый ответ и массовую обработку. На втором этапе, если уверенность недостаточна, транзакция передаётся в LLM-агент. Выбор данного подхода связан с необходимостью сократить число обращений к LLM и использовать генеративный компонент только для случаев, где контекстная интерпретация действительно оправдана.

Генерация рекомендаций автоматизирована через Financial Coach Agent. Агент не помещает полный набор сырых транзакций непосредственно в prompt. Вместо этого он использует инструменты анализа расходов, которые возвращают агрегированные данные: суммы по категориям, изменения относительно предыдущего периода, крупных мерчантов и возможные всплески расходов. Такой подход делает ответ более контролируемым и уменьшает риск произвольной генерации, не связанной с данными пользователя.

В границы прототипа также входит доставка результата. Core Service предоставляет REST API для получения транзакций и рекомендаций, а Notify Service может отправлять уведомления через Telegram. Экспериментальная оценка качества и эффективности выделена в отдельный этап работы и описывается в последующих подразделах после проведения измерений.

## Входные и выходные данные процесса

Входными данными для процесса являются события о финансовых операциях. В минимальном виде транзакция содержит transactionId, userId, сумму, описание, merchant name, MCC и timestamp. Эти поля используются как для ML-инференса, так и для формирования контекста при fallback-классификации.

Промежуточные данные появляются в ходе обработки. К ним относятся категория транзакции, значение confidence, источник классификации, статус транзакции, идентификаторы корреляции и события Kafka. Для рекомендаций промежуточными данными являются агрегаты расходов и запись запроса со статусом PENDING.

Выходом процесса является классифицированная транзакция и, при запуске второго контура, персональная финансовая рекомендация. Рекомендация сохраняется в PostgreSQL в виде структурированных данных и может быть получена через API или доставлена пользователю как уведомление.

## Критерии успешности автоматизации

Оценка успешности прототипа проводится на следующем этапе работы, однако критерии можно определить уже на уровне постановки задачи. Для классификации важны качество определения категории, доля случаев, ушедших в LLM fallback, и задержка обработки. Для агентного контура рекомендаций значимы корректность использования фактических данных, устойчивость структуры ответа и практическая полезность сформированного совета.

Отдельный критерий — трассируемость. В проекте используются transactionId и requestId, которые связывают входное событие, промежуточные Kafka-сообщения, записи в базе данных и логи LLM-вызовов. Это нужно для отладки, проверки спорных классификаций и дальнейшей экспериментальной оценки.

В результате задача 3.1 сводится к формализации бизнес-сценария, где простые операции обрабатываются быстрым ML-компонентом, а неоднозначные случаи и рекомендации передаются AI-агентам. Такой сценарий соответствует теме работы: он показывает интеграцию AI-агента в конкретный бизнес-процесс с понятными входами, выходами, ограничениями и критериями оценки, а не абстрактное применение LLM без связи с данными пользователя.
