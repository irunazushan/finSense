# 3.1. Постановка задачи, описание частично формализуемого бизнес-процесса

В рамках данной работы рассматривается задача автоматизации обработки финансовых транзакций и формирования персонализированных рекомендаций для пользователя финансового приложения. Такой сценарий выбран как пример бизнес-процесса, в котором часть операций хорошо описывается формальными правилами и структурами данных, а часть требует контекстной интерпретации. Именно эта неоднородность делает процесс удобным объектом для применения гибридного подхода: быстрый ML-компонент используется для первичной классификации, а AI-агент на базе LLM подключается к более сложным случаям.

Разработанный прототип FinSense ориентирован на два связанных процесса. Первый процесс связан с классификацией банковских транзакций по категориям расходов. Второй процесс использует уже классифицированную историю операций для генерации персональных финансовых рекомендаций. В реальном банковском или финтех-приложении эти процессы напрямую влияют на качество пользовательской аналитики: если транзакции отнесены к неверным категориям, рекомендации становятся менее полезными, а отчёты по расходам теряют практическую ценность.

## Общая постановка задачи

Основная задача прототипа состоит в построении системы, которая принимает поток транзакций, определяет категорию каждой операции и формирует рекомендации на основе накопленной истории расходов. При этом система должна учитывать, что не все транзакции одинаково просты для автоматической обработки. Часть операций имеет понятные признаки: сумма, MCC, merchant name, описание платежа. В таких случаях достаточно ML-классификатора, который выполняет быстрый inference через ONNX Runtime и возвращает категорию вместе с оценкой уверенности.

Другая часть транзакций содержит неоднозначные или слабо информативные описания. Например, merchant name может быть сокращён, транслитерирован, содержать технический идентификатор терминала или название платёжного посредника вместо фактического продавца. В таких случаях одной модели классификации может быть недостаточно: требуется учесть историю пользователя, соседние операции и смысловое содержание описания. Для этого в проекте используется Transaction Classifier Agent, который получает низкоуверенные случаи и обрабатывает их с помощью LLM.

Задача генерации рекомендаций имеет похожую природу. Агрегированные расходы по категориям можно вычислить детерминированно, но сам совет должен быть сформулирован с учётом поведения пользователя. Для этого Financial Coach Agent использует набор инструментов анализа расходов, получает фактические агрегаты из базы данных и передаёт их в LLM для генерации итоговой рекомендации.

## Описание бизнес-процесса

Обрабатываемый бизнес-процесс начинается с появления финансовой операции пользователя. В прототипе такие операции поступают в систему как события raw-transactions, что позволяет имитировать поток транзакций от внешней банковской системы или тестового генератора. Каждая транзакция содержит идентификатор, пользователя, счёт, сумму, описание, merchant name, MCC и дату операции.

После поступления события Core Service сохраняет транзакцию и запускает первичную классификацию. Для этого он синхронно обращается к Classifier Service, где выполняется ML-инференс через ONNX Runtime. Если возвращённое значение confidence достаточно высокое, результат классификации считается итоговым: транзакция получает категорию, источник классификации и статус завершённой обработки.

Если уверенность классификатора ниже заданного порога, Core Service не принимает результат как окончательный. Вместо этого он публикует событие в llm-classifier-requests. Transaction Classifier Agent получает запрос, формирует prompt на основе данных текущей транзакции и контекста пользовательской истории, вызывает LLM и публикует результат в llm-classifier-responses. После этого Core Service обновляет сохранённую транзакцию.

Отдельный поток связан с рекомендациями. Пользователь или планировщик инициирует запрос на получение финансового совета. Core Service создаёт запись рекомендации со статусом PENDING и публикует событие coach-requests. Financial Coach Agent читает транзакции пользователя, выполняет агрегирующие инструменты анализа и формирует рекомендацию с помощью LLM. Результат сохраняется в PostgreSQL, а событие coach-responses может быть использовано Notify Service для отправки уведомления.

В этом процессе Core Service выполняет роль оркестратора. Classifier Service отвечает за быстрый ML-контур, Transaction Classifier Agent — за контекстную дообработку сложных транзакций, Financial Coach Agent — за генерацию рекомендаций, а Kafka обеспечивает асинхронный обмен событиями между компонентами.

## Частично формализуемый характер процесса

Рассматриваемый процесс нельзя отнести ни к полностью формализованным, ни к полностью неформализованным. Его основа хорошо структурирована. У транзакции есть фиксированный набор полей, категории расходов задаются заранее, жизненный цикл обработки описывается конечным набором статусов, а взаимодействие сервисов строится на REST- и Kafka-контрактах. Эти элементы можно описать схемами, DTO, таблицами базы данных и состояниями.

При этом смысловая интерпретация финансовой операции не всегда сводится к значениям отдельных полей. Одна и та же строка описания может означать разные типы расходов в зависимости от merchant name, суммы, пользовательской истории и контекста. Новые продавцы, нестандартные названия, платёжные агрегаторы и неполные описания снижают качество прямой классификации. Здесь возникает слабо формализуемая часть процесса: системе нужно не только сопоставить признаки, но и интерпретировать их.

Генерация рекомендаций также имеет две стороны. Расходы по категориям, динамику за период и крупных мерчантов можно получить детерминированными запросами к данным. Но итоговый совет не должен быть простой строкой из шаблона. Он должен учитывать, где пользователь действительно тратит больше, какие категории изменились за период и какой вывод можно сделать на основе этих агрегатов. Поэтому в проекте LLM не заменяет аналитический расчёт, а используется после него, когда фактические данные уже подготовлены инструментами.

Такое разделение отражает инженерную логику прототипа. Формализуемые части процесса реализованы через сервисные контракты, базы данных, ML-инференс и обработку событий. Слабо формализуемые фрагменты вынесены в AI-агенты, которые работают асинхронно и не блокируют основной поток обработки.

## Границы автоматизации в прототипе

В прототипе автоматизируется полный путь от поступления транзакции до получения классифицированной операции и рекомендации. Система не ограничивается отдельным ML-классификатором: она включает контур fallback-классификации и отдельный агентный контур финансового советника.

Классификация транзакций выполняется в два этапа. На первом этапе применяется ML/ONNX-компонент, рассчитанный на быстрый ответ и массовую обработку. На втором этапе, если уверенность недостаточна, транзакция передаётся в LLM-агент. Такой подход снижает количество обращений к LLM и оставляет генеративный компонент для тех случаев, где он действительно полезен.

Генерация рекомендаций автоматизирована через Financial Coach Agent. Агент не работает напрямую с полным набором сырых транзакций в prompt. Вместо этого он использует инструменты анализа расходов, которые возвращают агрегированные данные: суммы по категориям, изменения относительно предыдущего периода, крупных мерчантов и возможные всплески расходов. Это делает ответ более контролируемым и уменьшает риск произвольной генерации, не связанной с данными пользователя.

В границы прототипа также входит доставка результата. Core Service предоставляет REST API для получения транзакций и рекомендаций, а Notify Service может отправлять уведомления через Telegram. При этом экспериментальная оценка качества и эффективности выделена как отдельный этап работы и будет описана в последующих подразделах после проведения измерений.

## Входные и выходные данные процесса

Входными данными для процесса являются события о финансовых операциях. В минимальном виде транзакция содержит transactionId, userId, сумму, описание, merchant name, MCC и timestamp. Эти поля используются как для ML-инференса, так и для формирования контекста при fallback-классификации.

Промежуточные данные появляются в ходе обработки. К ним относятся категория транзакции, значение confidence, источник классификации, статус транзакции, идентификаторы корреляции и события Kafka. Для рекомендаций промежуточными данными являются агрегаты расходов и запись запроса со статусом PENDING.

Выходом процесса является классифицированная транзакция и, при запуске второго контура, персональная финансовая рекомендация. Рекомендация сохраняется в PostgreSQL в виде структурированных данных и может быть получена через API или доставлена пользователю как уведомление.

## Критерии успешности автоматизации

Оценка успешности прототипа будет проводиться на следующем этапе работы, однако критерии можно определить уже на уровне постановки задачи. Для классификации важны качество определения категории, доля случаев, ушедших в LLM fallback, и задержка обработки. Для агентного контура рекомендаций важны корректность использования фактических данных, устойчивость структуры ответа и практическая полезность сформированного совета.

Отдельным критерием является трассируемость. В проекте используются transactionId и requestId, которые позволяют связать входное событие, промежуточные Kafka-сообщения, записи в базе данных и логи LLM-вызовов. Это необходимо не только для отладки, но и для дальнейшей экспериментальной оценки.

В результате задача 3.1 сводится к формализации бизнес-сценария, где простые операции обрабатываются быстрым ML-компонентом, а неоднозначные случаи и рекомендации передаются AI-агентам. Такой сценарий соответствует теме работы, поскольку демонстрирует не абстрактное применение LLM, а интеграцию AI-агента в конкретный бизнес-процесс с понятными входами, выходами, ограничениями и критериями оценки.
